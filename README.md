## Case 1. Avalanche NFT 민팅 서버(Dapp) 1인 개발

- 기간: **2025.10.27 ~ 2025.12.31**
- 형태: 회사 내부 신규 DApp 서버 MVP 개발
- 역할: **기능 정의, 서버 설계, 실제 개발, 인프라 구축, 배포 구조 정리까지 전반 담당**
- 기술 스택: **Node.js, TypeScript, NestJS, Web3.js, AWS, Terraform**
- 참고 자료: [miji-dapp-server 가이드 문서](https://mijiworld.github.io/miji-dapp-server/)

### 프로젝트 개요

회사 내부에서 Avalanche 기반의 신규 DApp 서버가 필요해졌고, 별도 기획자나 큰 서버팀이 붙어 있는 구조가 아니었습니다.  
그래서 요구사항을 구체화하는 단계부터 시작해, 서버 구조를 잡고, 필요한 API와 DApp 연동 로직을 구현하고, AWS 환경에 Terraform 기반으로 인프라를 올려 **MVP 수준의 서버를 실제 동작 가능한 상태로 완성**하는 것이 목표였습니다.

저는 이 프로젝트를 **사실상 1인 서버 개발** 형태로 진행했습니다.

### 제가 맡은 역할

작은 팀 환경이었기 때문에 역할이 좁게 나뉘지 않았고, 아래 범위를 전체적으로 맡았습니다.

- 신규 서버 기능 정의 및 브레인스토밍
- API 및 서버 구조 설계
- NestJS 기반 실제 서버 개발
- Web3.js 기반 DApp 연동 로직 구현
- 기술 문서 작성 및 공유
- Terraform 기반 AWS 인프라 구성
- CI/CD가 가능한 수준의 배포 구조 정리

정리하면, 단순 구현자가 아니라 **“무엇을 만들지 정하는 단계”부터 “실제로 서비스 형태로 올리는 단계”까지** 전체를 담당했습니다.

### AI를 어떻게 활용했는가

이 프로젝트에서는 AI를 한두 군데에 보조적으로 쓴 것이 아니라, 개발 사이클 전반에 적극적으로 사용했습니다.

#### 1) 초기 기능 정의와 브레인스토밍

프로젝트 초반에는 요구사항이 구체적으로 정리되어 있지 않았기 때문에, 필요한 서버 기능과 API 흐름, 외부 의존성, 운영 시나리오를 빠르게 정리해야 했습니다.

이 단계에서 ChatGPT, Gemini, Perplexity 등을 활용해 아래와 같은 작업을 반복했습니다.

- DApp 서버에서 필요한 핵심 기능 목록 정리
- 민팅/검증/지갑 연동 흐름 브레인스토밍
- 우선 구현해야 하는 MVP 범위 도출
- 백로그 초안 구성
- 의사결정 전에 고려할 리스크 항목 정리

이 과정에서 AI는 “답을 대신 내는 존재”라기보다는, **제가 생각한 구조를 빠르게 넓히고 비교하게 해주는 토론 상대**에 가까웠습니다.

#### 2) 개발 전 기술 문서 작성

실제 개발에 들어가기 전에, 다른 사람들과 공유할 문서를 먼저 만드는 과정에서도 AI를 많이 활용했습니다.

예를 들어,

- 서버에 필요한 기능 목록 정리
- API/모듈 구조 초안 설명
- 작업 예정 내역 문서화
- 개발 순서 정리
- 인프라 방향성 설명

같은 문서를 작성할 때 AI로 초안을 만들고, 이후 제가 서비스 맥락과 실제 제약사항에 맞게 수정했습니다.

이 방식은 문서 작성 시간을 줄여준 것 이상으로, **개발 착수 전에 내 생각을 구조화하는 데 큰 도움**이 됐습니다.

#### 3) 실제 코드 작성

실제 구현 단계에서는 **Cursor + GPT Codex** 조합을 가장 적극적으로 사용했습니다.

AI를 활용한 대표적인 영역은 다음과 같습니다.

- NestJS 모듈/서비스/DTO 초안 생성
- 반복적인 보일러플레이트 코드 작성
- Web3 연동부 초안 생성
- 예외처리 구조 제안
- 테스트 코드 초안 작성
- 리팩토링 후보 탐색

특히, “이 코드를 통째로 맡긴다”기보다는,

- 제가 먼저 구조와 방향을 정하고
- AI에게 한정된 범위의 초안을 빠르게 받거나
- 기존 코드를 기준으로 개선안을 제안받고
- 최종 반영 여부는 직접 검토

하는 방식으로 사용했습니다.

즉, AI는 속도를 올려주는 도구였지만, **구조와 품질의 책임은 제가 직접 쥐고 진행**했습니다.

#### 4) Terraform 기반 인프라 구축

인프라는 AWS에 Terraform 기반으로 올렸습니다.

이 부분도 큰 틀과 방향은 제가 정한 뒤, 실제 Terraform 코드 작성에는 AI를 적극 활용했습니다.

진행 방식은 대략 아래와 같았습니다.

- 필요한 AWS 자원과 구성 방향은 제가 설계
- Terraform 초안은 AI에게 생성 요청
- `terraform plan` 을 반복 수행하며 실제 차이를 검증
- AI가 부족하게 만든 부분은 다시 구체적으로 지시하여 수정
- dev 환경에서 확인 후 구조를 다듬음

결국 **인프라 설계는 사람이 주도하고, 구현 반복은 AI로 가속**하는 형태였습니다.

### AI의 한계와 실제로 겪은 문제

이 프로젝트에서 가장 크게 느낀 부분은, **AI는 최신 기술 변화나 도메인 특수성을 개발자가 제대로 잡아주지 않으면 쉽게 낡은 방향으로 흘러간다**는 점이었습니다.

예를 들어, 중간에 Coinbase Wallet 지원과 관련해 컨트랙트 월렛 스펙 및 FIDO 계열 표준과 맞물리는 이슈를 다뤄야 하는 상황이 있었습니다. 이 부분은 2025년 말 기준으로도 비교적 최근 흐름이 반영된 영역이었는데, 사용자가 사전지식 없이 “알아서 구현해줘” 식으로 두루뭉술하게 요청하면 AI는 자기가 익숙한 **몇 년 이상 된 블록체인 트렌드**를 기준으로 답을 내놓는 경우가 있었습니다.

즉,

- 최신 기술 맥락이 반영되지 않거나
- 이미 달라진 지갑/계정 모델을 과거 기준으로 설명하거나
- 실제 문서가 아닌 일반론 위주로 코드를 제안하는 문제

가 있었습니다.

이때 중요한 것은, **개발자가 전문지식과 최신 문서 기준을 명확히 쥐고 AI를 조율하는 것**이었습니다.

제가 실제로 취한 방식은 다음과 같았습니다.

- 관련 공식 문서나 최신 스펙을 먼저 확인
- AI에게 구체 문서 기준으로 다시 작업 요청
- 두루뭉술한 요구 대신 범위를 좁혀서 지시
- 결과물을 그대로 신뢰하지 않고 직접 검증

이 경험을 통해, AI 활용 역량은 단순히 “잘 시키는 법”보다도 **무엇이 틀렸는지 알아차릴 수 있는 개발자의 기본기와 전문성** 위에서 성립한다는 점을 다시 확인했습니다.

### 성과

이 프로젝트의 가장 큰 성과는, 구체화되지 않은 상태에서 시작한 신규 서버 개발을 **짧은 기간 안에 MVP 수준으로 완성하고, 자동 배포와 테스트까지 가능한 형태로 정리했다**는 점입니다.

예전처럼 AI 없이 전부 수작업 중심으로 진행했다면 더 긴 개발 기간을 잡아야 했겠지만, 이번에는 아래와 같은 부분에서 속도와 밀도를 높일 수 있었습니다.

- 초기 요구사항 정리 속도 향상
- 문서화와 설계 초안 작성 시간 단축
- 반복 구현 속도 향상
- Terraform 작성 및 수정 반복 시간 단축
- 테스트/배포 구조 정리 속도 개선

결과적으로, **아무 것도 구체화되지 않은 단계에서 출발한 서버를 1인 개발에 가까운 형태로 MVP까지 완결**해낼 수 있었습니다.

### 이 사례에서 제가 강조하고 싶은 점

이 프로젝트에서 AI는 단순히 코드를 몇 줄 대신 써준 도구가 아니었습니다.

저는 AI를 다음과 같이 사용했습니다.

- 기획 공백을 메우는 정리 도구
- 설계 논의를 빠르게 확장하는 파트너
- 반복 구현을 줄여주는 개발 가속 장치
- 문서화 생산성을 높이는 도구
- 인프라 작업의 반복 비용을 줄이는 도우미

하지만 동시에,

- 최신성 확인
- 서비스 맥락 반영
- 예외처리와 품질 책임
- 최종 의사결정

은 반드시 사람이 맡아야 한다는 것도 실무에서 분명하게 경험했습니다.

## Case 2. VTuber LLM 유틸리티 프로젝트

- 형태: 개인/팀 기반 인디 프로젝트
- 팀 구성: **3인 (저: 리더 + 서버 개발 / Unity 개발자 / 디자이너)**
- 사용 기술: **Unity, Python, Notion, Figma**
- 참고 자료:
  - [렛플 프로젝트 소개](https://letspl.me/project/2560/shortcut)
  - [진행 중인 Python 서버 저장소](https://github.com/sinbumu/vtuber_llm_sep_server)
  - [원본 오픈소스](https://github.com/Open-LLM-VTuber/Open-LLM-VTuber)

### 프로젝트 목표

기존에 공개된 `Open-LLM-VTuber` 계열 오픈소스는 기능적으로 흥미롭지만, 일반 사용자가 쓰기에는 개발자 친화적인 요소가 많아 진입장벽이 높다고 느꼈습니다.

그래서 저는,

> **비개발자도 LLM 서비스 API Key만 준비하면 비교적 쉽게 사용할 수 있는 VTuber/캐릭터형 유틸리티 툴을 만들고, 궁극적으로는 Steam에 배포해보자**

라는 방향으로 접근했습니다.

즉, 단순한 데모나 오픈소스 실행 실험이 아니라, **일반 사용자가 실제로 사용할 수 있는 제품 형태로 다듬는 것**이 목표입니다.

### 현재 진행 방식

현재는 제가 팀 리더 역할을 맡아, 매주 주간회의를 진행하면서 전체 방향을 정리하고 각자 원격으로 개발하고 있습니다.

- 저는 서버/기술 방향 및 전체 리딩 담당
- Unity 개발자는 클라이언트/실행 환경 담당
- 디자이너는 시각 요소 및 사용자 경험 관련 작업 담당

기획과 협업은 Notion, Figma 등을 활용해 진행 중입니다.

### AI를 어떻게 활용했는가

이 프로젝트에서도 AI는 단순 보조가 아니라, **오픈소스 분석 → 구조 변경 → 기능 아이데이션 → 실제 개발** 전반에 들어와 있습니다.

#### 1) 오픈소스 코드베이스 분석

처음에는 원본 오픈소스의 구조를 파악하는 것 자체가 중요한 과제였습니다.

AI를 활용해,

- 주요 모듈의 역할 요약
- 서버 실행 흐름 파악
- 어떤 부분이 Unity 연동에 걸림돌이 되는지 정리
- 분리 가능한 기능과 불필요한 기능 구분

을 빠르게 진행했습니다.

이 과정은 특히, 이미 규모가 있는 Python 프로젝트를 처음 읽을 때 **“어디부터 뜯어봐야 하는지”를 잡는 데 큰 도움**이 됐습니다.

#### 2) 포크 이후 필요한 부분만 떼어내는 작업

현재는 해당 프로젝트를 포크한 뒤, 필요한 부분만 남기고 **Unity 환경에서 `.exe` 형태로 실행 가능한 Python 서버**로 개조하는 방향으로 작업 중입니다.

이 과정에서 AI를 활용한 영역은 다음과 같습니다.

- 기존 구조에서 유지할 모듈 / 제거할 모듈 구분
- 실행 진입점 단순화 아이디어 정리
- 환경 의존성 축소 방안 검토
- Python 코드 수정 초안 생성
- 기능 추가 시 영향 범위 분석

즉, AI는 이 프로젝트에서 “처음부터 새로 만드는 개발”보다도, **기존 오픈소스를 제품화 가능한 구조로 변형하는 과정**에서 특히 유용하게 작동했습니다.

#### 3) 추가 기능 설계와 구현 아이디어

기존 오픈소스에는 대화가 길어질 경우 맥락 관리가 비교적 단순한 편이었습니다. 오래된 대화가 누적되더라도, 일반 사용자 관점에서 자연스럽게 유지·정리하는 메커니즘이 부족하다고 느꼈습니다.

그래서 현재는,

- 반드시 유지해야 하는 페르소나/핵심 컨텍스트는 남기고
- 최근 대화 일부는 유지하되
- 오래된 대화는 일정량이 쌓이면 요약하거나 정리하는 방식

의 컨텍스트 관리 기능을 추가하려고 고민 중입니다.

이 부분에서도 AI를 활용해,

- 메모리 구조 아이디어 비교
- 대화 이력 요약 방식 검토
- 유지해야 할 컨텍스트와 버려도 되는 히스토리 구분 기준 논의
- 구현 접근 방식 초안 생성

을 진행하고 있습니다.

아직 진행 중인 기능이지만, 이 작업은 단순한 UI 개선이 아니라 **실제 LLM 기반 대화 품질과 사용자 경험을 좌우하는 핵심 서버 기능**이라고 생각하고 있습니다.

### AI를 활용하며 느낀 점

이 프로젝트는 회사 실무 사례와는 또 다른 의미가 있었습니다.

실무에서는 보통 일정과 요구사항이 먼저 있고 AI를 생산성 도구로 활용하는 느낌이 강했다면, 이 프로젝트에서는 오히려 **“AI와 함께 어떤 제품을 만들 수 있을까”를 탐구하는 실험성**이 더 큽니다.

특히 느낀 점은 다음과 같습니다.

- AI는 낯선 오픈소스 구조를 빠르게 이해하는 데 강하다.
- 기존 프로젝트를 뜯어고치는 작업에서도 방향 설정에 도움이 된다.
- 다만 비동기 흐름, 상태 관리, 프로젝트 고유 컨벤션은 여전히 사람이 세밀하게 봐야 한다.
- 제품화 수준으로 가려면, “작동하는 코드”보다 “사용자에게 안정적으로 전달되는 구조”가 더 중요하다.

### 현재 상태

현재는 팀 단위로 원격 개발을 이어가고 있으며, 주간회의를 통해 방향을 맞추고 있습니다.  
Python 서버 저장소는 공개되어 있고, README 기준으로 로컬 환경에서 실행 가능한 형태로 관리 중입니다.

아직 완성형 출시 단계는 아니지만, 이 프로젝트는 제가 AI를 활용해 **오픈소스를 분석하고, 제품화 가능성을 검토하고, 팀 단위로 실제 구현까지 연결하는 방식**을 보여주는 사례라고 생각합니다.

## 사용한 AI 도구

최근 실제로 자주 사용한 도구는 아래와 같습니다.

- **ChatGPT**: 브레인스토밍, 구조 정리, 문서 초안, 코드 보조
- **Cursor**: 실제 코드 수정/생성 및 리팩토링 작업
- **GPT Codex**: 최근 프로젝트에서 Cursor 위주 개발 시 주력으로 활용
- **Gemini**: 비교 관점의 아이디어 탐색 및 초안 검토
- **Perplexity**: 기술 흐름이나 관련 자료 탐색 보조
- **Claude**: Cursor 환경에서 코드 보조에 자주 활용했으나, 최근엔 GPT Codex 비중이 더 높음

제 작업 방식은 보통 하나의 AI에 전적으로 의존하기보다,

- 아이디어 정리는 ChatGPT/Gemini
- 코드 작업은 Cursor
- 자료 탐색은 Perplexity

처럼 역할을 나누어 사용하는 편입니다.

## 마무리

저는 AI를 “개발자를 대체하는 존재”로 보지 않습니다.  
대신, **좋은 개발자가 더 넓게 사고하고, 더 빨리 검증하고, 반복 작업을 줄이며, 본질적인 판단에 더 집중하게 해주는 파트너**에 가깝다고 생각합니다.

실제로 저는,

- 실서비스 DApp 서버를 1인에 가깝게 개발하면서 AI를 적극 활용해봤고
- 오픈소스 기반 LLM 프로젝트를 제품화 관점에서 다시 뜯어보는 실험도 하고 있으며
- 모델 학습/발표/개인 프로젝트 차원에서도 AI 관련 시도를 꾸준히 이어가고 있습니다.

코코네가 말하는 AX Centric Development는 단순히 “AI 툴을 써봤다”가 아니라, **AI를 개발 문화와 문제 해결 방식 안으로 실제로 끌어들여 본 경험**이 중요하다고 이해하고 있습니다.

저 역시 그 방향에 강한 흥미가 있고, 앞으로도 계속 실험하고 개선해보고 싶습니다.
